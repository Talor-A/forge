
Your instincts are right, and the good news is that the tooling for treating ML like real software engineering has gotten quite good in the last few years. Most ML tutorials are optimized for someone exploring an idea once, not for someone shipping a system, so they're a bad model. Here's how to bring your existing practices over.
Fast feedback by miniaturizing everything. Maintain a "tiny" config of your training pipeline that runs end-to-end in under a minute: tiny model (a few layers, hidden dim 8), tiny batch, tiny dataset (a single batch held in memory), 2 optimization steps. This is your smoke test, and it should run in CI on every push. Most of your bugs — wrong shapes, broken data pipeline, autograd not flowing, eval/train mode mistakes, checkpoint round-tripping — surface here without needing a GPU. Separately, the single most useful diagnostic in supervised ML is overfit a single batch: if your model can't drive loss to ~0 on one batch with a fixed input, the model or loss is broken and no amount of bigger-data training will save you. Make that an explicit test.
Pin down tensor shapes and dtypes. This is the biggest leverage point if you're coming from strong typing. Use jaxtyping (works with PyTorch, JAX, NumPy, TF despite the name) to annotate tensor shapes in function signatures: Float[Tensor, "batch seq d_model"]. Combine with beartype or typeguard and you get runtime shape checking on every call. This catches the class of bug where a silent broadcast turns (B, 1) into (B, B) and your loss is quietly wrong. Pyright/pyrefly on top of that handles the rest of your typing. For configs, use Pydantic or dataclasses with Hydra — never untyped dicts.
Make invariants explicit and cheap. Sprinkle assertions liberally; they're free at training scale. Useful ones: loss is finite, grad norms are bounded and non-zero, action probabilities sum to 1, value estimates aren't drifting toward NaN, advantages have ~0 mean after normalization, replay buffer indices are valid, env observations are within declared bounds. For RL specifically: hand-compute the discounted return and GAE on a 3-step toy trajectory and unit-test your implementation against it — these formulas are where almost everyone has a subtle off-by-one or wrong-axis bug. Also test the env in isolation (reset returns valid obs, step is deterministic given a seed, terminal flag behaves correctly) before plugging it into a learner.
Kill hidden state. This is the source of most "works on my machine" pain in ML. Treat checkpoints as explicit inputs and outputs of pure functions, never as ambient state. If you use notebooks, either restart-and-run-all as your only acceptable execution mode, or convert with jupytext and run as scripts. Seed everything (random, numpy, torch, torch.cuda, env), log the seed, and have a determinism test that runs two short trainings with the same seed and asserts the resulting weights are bitwise (or near-bitwise — CUDA nondeterminism is a thing) identical. Have a separate test that saves a checkpoint, loads it, and checks the model produces identical outputs.
Reproducibility infrastructure. Use uv or pixi for locked Python environments — fast and modern, none of the conda pain. Every experiment should log: git SHA (and a dirty flag), full resolved config, seed, hardware, library versions. Weights & Biases, MLflow, or Aim all do this; pick one. If a run can't be reconstructed from its log, it didn't happen. For data, hash your dataset and pin the hash; data drift is a silent reproducibility killer.
RL-specific advice, since you mentioned it. RL has uniquely long and noisy feedback loops, so lean even harder on layered testing. Verify your algorithm on a tiny tabular environment with a known optimum (gridworld, FrozenLake) before scaling up — if PPO can't solve CartPole reliably across seeds, it won't solve your real env. Always run with multiple seeds and plot the spread; a single seed is a coin flip and tells you almost nothing. Diff against a reference implementation (CleanRL is excellent for this — single-file, readable, and known-correct) when something seems off; a line-by-line comparison usually finds the bug within an hour. Log per-component diagnostics: policy entropy, value loss, explained variance, KL divergence between old and new policy, gradient norms. When training diverges, these tell you which part broke.
The general pattern. Build bottom-up with each layer testable: env → data pipeline → model forward → loss → optimizer step → training loop → full run. Each gets unit tests with shape/dtype/value assertions. CI runs the fast tests plus the smoke-test training. Long runs are launched only after the smoke test passes. Your "did it work" question is answered by automated checks on metrics, not by squinting at a tensorboard plot.
A couple of resources that are unusually good if you want deeper reads: Andrej Karpathy's "Recipe for Training Neural Networks", Joel Grus's "I Don't Like Notebooks" talk, and the CleanRL codebase for RL specifically.
